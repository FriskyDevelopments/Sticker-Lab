# Brand Architecture

## Purpose

This ecosystem works best when the three names are not treated as synonyms. Each one carries a different job, a different emotional weight, and a different point of contact with the user.

The architecture is intentionally simple:

- **Frisky Developments** = creator / studio / personality / giver
- **STIX MΛGIC** = engine / magic / expressive system
- **LORE** = identity layer / profile layer / user-facing presence

## Core relationship

The clearest way to describe the system is:

- **Frisky gives the magic.**
- **STIX is the magic.**
- **LORE gives that magic identity.**

This creates a clean separation between authorship, capability, and presence.

## Role definitions

### Frisky Developments
Frisky Developments is the parent creative studio and personality layer above the products. It should feel like the signature behind the experience rather than a corporate wrapper around it.

Frisky is where warmth, intention, generosity, and slight mischief live. It is the source of tone. It signals that the ecosystem is made by someone with taste, care, and a point of view.

### STIX MΛGIC
STIX MΛGIC is the primary product magic: the engine, transformation system, and expressive behavior that users feel in motion. It should read as the active force inside the experience rather than the company behind it.

STIX carries the strongest product energy. It is the hero mark when the interface is performing, transforming, or demonstrating the system's expressive capability.

### LORE
LORE is the identity-facing layer. It is where the user's presence, profile, memory, and expression become legible. If STIX is the magic engine, LORE is the layer that helps the magic belong to someone.

LORE should feel closer to selfhood and continuity than to infrastructure.

## Practical usage rules

### When to lead with Frisky Developments
Use Frisky Developments in:

- about pages and studio descriptions
- ecosystem overviews
- partnership or creator-facing documentation
- parent-brand signatures in footers, credits, and brand systems
- tone-setting material that introduces the ethos behind the products

Do **not** use Frisky as the default label for product features, engine behaviors, or profile mechanics.

### When to lead with STIX MΛGIC
Use STIX MΛGIC in:

- product UI and motion systems
- engine descriptions
- transformation workflows
- interaction design discussions
- system behavior and visual-performance guidance

STIX should remain the main product mark, especially where the magic is being experienced directly.

### When to lead with LORE
Use LORE in:

- identity and profile features
- user-presence systems
- memory, collection, or self-expression surfaces
- settings or representations of a user's ongoing presence

LORE should not be used as a substitute for the parent studio or the transformation engine.

## Messaging hierarchy

A useful narrative order is:

1. **Frisky Developments** introduces the ecosystem with character.
2. **STIX MΛGIC** demonstrates the system's expressive power.
3. **LORE** personalizes that power for the user.

This order keeps the ecosystem legible without making Frisky feel corporate or making STIX carry every conceptual burden alone.

## What was missing in the repo audit

Current documentation strongly defines STIX as a product/system, but it does not yet establish:

- a parent-brand role for Frisky Developments
- a clear explanation of how LORE fits into the system
- consistent language for the relationship between the studio, the engine, and the identity layer
- guidance for when Frisky should appear versus when it should stay in the background

This document fills that structural gap so future copy, identity, and implementation work can stay aligned.
